# PLAN

## GOAL

`ozcar` needs a canonical initial slice that turns the sibling-repo architecture research into an executable build contract. The immediate next slice is Phase 1: establish the TypeScript CLI foundation, durable run-store primitives, and the target `src/` layout so later phase orchestration can land without reshaping the repo.

This repo now uses the POP-managed local artifact surface. `PLAN.md` remains canonical; this artifact records the prompt-authoring handoff for the first implementation slice and points later work at the local execution and review templates.

## SCOPE

### In Scope

- canonical `PLAN.md` and `AGENTS.md` aligned to the filesystem-first audit runner design
- the Phase 1 implementation slice for toolchain bootstrap, CLI command skeletons, typed contracts, and run-store primitives
- translation of `~/oz/...` research references into parent-relative paths from this repo, such as `../fuzzing-team-foz/...` and `../apprentice-victor/...`
- later implementation and review compatibility with `docs/execs/` and `docs/reviews/`

### Out of Scope

- implementing Phases 2 through 5 in this turn
- a public plugin system, multiple audit packs, or duplicated helper layouts
- shell/Python core orchestration or markdown-only machine contracts

## CONSTRAINTS

- `PLAN.md` is canonical; `docs/plans/*.md` are records
- follow `.agents/skills/artifact-md-workflow/SKILL.md` plus the local templates for persisted work
- keep the core runtime TypeScript-only
- keep machine contracts JSON-first and generate Markdown from validated findings only
- keep the first implementation slice limited to Phase 1
- treat sibling repos under `../fuzzing-team-foz/` and `../apprentice-victor/` as read-only references

## PHASES

### Phase 1: Workspace Foundation And Run Store

Objective:
Establish the TypeScript workspace, CLI entrypoint, durable run-store contract, and typed module skeleton.

Deliverables:
- runnable TypeScript toolchain and validation scripts
- `src/cli.ts`
- command, contract, and store skeletons for the v1 layout
- tests for run creation, lock behavior, and event persistence

### Phase 2: Plan And Scan

Objective:
Add provider selection plus gated `plan` and `scan` phases.

Deliverables:
- provider adapters
- prompt sources
- phase implementations and gates

### Phase 3: Triage, Validate, And Summarize

Objective:
Gate findings into validated JSON and generate reports mechanically.

Deliverables:
- triage and validation phases
- report builders
- deterministic summary rebuild path

### Phase 4: Operational Commands And Replay

Objective:
Complete the operational CLI and safe replay/rebuild/doctor flows.

Deliverables:
- full command wiring
- replay support
- doctor checks

### Phase 5: Verification Hardening

Objective:
Lock down seam behavior with fixtures and replayable sample runs.

Deliverables:
- component tests
- parser/dedup fixtures
- end-to-end run fixture

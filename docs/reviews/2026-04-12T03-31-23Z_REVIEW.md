# REVIEW

## VERDICT
PARTIAL
The current `ozcar` runtime is solid as a TypeScript-only, filesystem-first dry-run audit pipeline, but it is not yet ergonomic for real OAuth-backed auditing because provider resolution stops at CLI detection and the scan path remains `--dry-run` only.

- Dry-run contract evidence: [run.ts](/Users/x/oz/ozcar/src/commands/run.ts#L18) exposes only a dry-run operational seam and rejects non-`--dry-run` execution at [run.ts](/Users/x/oz/ozcar/src/commands/run.ts#L43), while [scan.ts](/Users/x/oz/ozcar/src/phases/scan.ts#L238) still synthesizes `output.json` from fixture-backed findings instead of dispatching a real provider call.
- Provider boundary evidence: [base.ts](/Users/x/oz/ozcar/src/providers/base.ts#L57) resolves providers by PATH detection and default model selection only, and the built-in provider definitions at [codex.ts](/Users/x/oz/ozcar/src/providers/codex.ts#L3) and [claude.ts](/Users/x/oz/ozcar/src/providers/claude.ts#L3) contain metadata only. There is no provider execution surface analogous to a live `run_agent`.
- Durable-state strength evidence: [triage.ts](/Users/x/oz/ozcar/src/phases/triage.ts#L32) and [validate.ts](/Users/x/oz/ozcar/src/phases/validate.ts#L37) write per-finding JSON sidecars behind deterministic gates, and [run-health.ts](/Users/x/oz/ozcar/src/store/run-health.ts#L50), [status.ts](/Users/x/oz/ozcar/src/commands/status.ts#L6), and [doctor.ts](/Users/x/oz/ozcar/src/commands/doctor.ts#L6) give the project a strong resumability and artifact-health surface once a run exists.
- Validation rerun: `npm run build` passed, `npm test` passed (`6` files, `37` tests), `python3 -B -m unittest discover -s tests` passed in `../auditor-bench` (`48` tests), and a local `node dist/cli.js run ... --dry-run --finding-fixture ...` demo produced healthy `status` and `doctor` output with replayable plan, scan, triage, validation, and report artifacts.

## BUGS
None

## GAPS
- Live provider execution is not implemented. A user with a valid Codex or Claude OAuth-backed CLI can be detected, but cannot yet run an actual provider-backed audit through `ozcar` because the only supported execution path is dry-run artifact synthesis from fixtures and deterministic heuristics.
- Ergonomic onboarding is missing. The repo has no top-level README or quickstart, and the exposed CLI in [cli.ts](/Users/x/oz/ozcar/src/cli.ts#L23) expects a low-level flag set (`--scope`, `--scan-target`, `--research-direction`, optional fixtures) rather than a single “audit this repo” workflow with documented prerequisites.
- Benchmark-facing export is not explicit. Validated results are currently persisted as per-finding directories with [validate.ts](/Users/x/oz/ozcar/src/phases/validate.ts#L64) through [validate.ts](/Users/x/oz/ozcar/src/phases/validate.ts#L115), but there is no stable benchmark export command or canonical aggregated findings JSON for downstream harnesses to consume without knowing internal directory structure.

## ANTI-PATTERNS
- The current provider abstraction is metadata-only. It cleanly models provider names, defaults, and detection, but it does not own the actual execution or auth/session boundary, so the main product promise (“run this with Codex or Claude”) still lives outside the abstraction rather than behind it.

## IMPROVEMENTS
- Add a small provider runner contract that shells out to authenticated `codex` and `claude` CLIs, captures raw prompt and response artifacts, and lets `plan` and `scan` move past synthetic dry-run outputs without reshaping the durable store.
- Add a top-level README with one minimal install/login/first-run path and pair it with a thinner CLI entrypoint that can infer scope defaults from the target repo instead of requiring repeated manual flags.
- Add a benchmark-friendly export surface, for example `ozcar export-findings --run-id <id>`, that emits normalized JSON or Markdown directly from validated bundles so external tools do not need to scrape internal gate directories.

# REVIEW

## VERDICT
PARTIAL
The Pi-native core is small, coherent, and well-tested against the current contract, but the repo still has one correctness bug plus two material ergonomics gaps before it matches current Pi extension/package best practices for reusable audit work.
Compared against the checked-in repo contract and current upstream Pi extension/package patterns, `npm run build` and `npm test` both pass, yet the current workflow still depends on a collision-prone auto audit id, an LLM-only checkpoint surface, and CLI-first packaging that keeps cross-project use less ergonomic than it should be.

## BUGS
- [.pi/extensions/ozcar/state/labels.ts](/Users/x/oz/ozcar/.pi/extensions/ozcar/state/labels.ts#L26) derives the default `auditId` by truncating the normalized focus string to 64 characters with no disambiguator. The default quickstart flow still tells users to start audits with `/ozcar-audit-start <focus>` in [README.md](/Users/x/oz/ozcar/README.md#L13), and exports are keyed only by `auditId` under `.ai-auditor/audits/<audit-id>/`. Two long or similarly-prefixed focuses can therefore alias to the same artifact root and overwrite each other's durable output even though the user never chose the same audit id explicitly.

## GAPS
- The durable Phase 4 -> Phase 6 checkpoint step is still agent-only. The primary user docs tell humans to "use the `ozcar_store_audit_snapshot` tool" in [README.md](/Users/x/oz/ozcar/README.md#L17), but the repo exposes no companion slash command or file-backed adapter for that step; the only checkpoint entrypoint is the LLM tool registered in [.pi/extensions/ozcar/tools/audit-artifact-snapshot.ts](/Users/x/oz/ozcar/.pi/extensions/ozcar/tools/audit-artifact-snapshot.ts#L12). That makes the most important persistence seam awkward to drive manually, awkward to script, and harder to verify outside agent turns.
- The repo is not yet packaged as a reusable Pi package. [package.json](/Users/x/oz/ozcar/package.json#L5) still exposes only the transition CLI `bin` and does not declare a `pi` manifest or package-style resource roots, even though current Pi package guidance expects either a `pi` manifest or top-level `extensions/`, `skills/`, and `prompts/` directories. As a result, `ozcar` is easy to use only from this checkout or by passing the raw extension file path, not through the more ergonomic `pi install /path/to/ozcar` or `pi -e /path/to/ozcar` package workflow that current Pi packages use.

## ANTI-PATTERNS
- The package surface still teaches the legacy interface first. [package.json](/Users/x/oz/ozcar/package.json#L5) and [src/cli.ts](/Users/x/oz/ozcar/src/cli.ts#L23) present `ozcar run/resume/replay/...` as the package entrypoint even though [README.md](/Users/x/oz/ozcar/README.md#L5) and `PLAN.md` say Pi is the real product surface now. Current Pi extension packages such as the upstream `pi-boomerang` and `pi-context` patterns expose the extension/package surface directly instead of shipping a separate legacy-first command tree as the default package identity. Keeping both surfaces live is defensible during transition, but leaving the CLI as the package face is unnecessary split-brain design and raises the maintenance cost of every future audit improvement.

## IMPROVEMENTS
- Add a human-facing checkpoint command such as `/ozcar-audit-checkpoint <snapshot.json>` or `/ozcar-audit-save`, and keep `ozcar_store_audit_snapshot` as the tool backend. That would make the export seam easy to test by hand, easy to script, and still natural for the model to call.
- Make default audit ids collision-resistant. Keep the explicit `<audit-id> :: <focus>` form, but for focus-only starts append a short stable hash or timestamp suffix instead of truncating to the first 64 slug characters.
- Add Pi package metadata to [package.json](/Users/x/oz/ozcar/package.json#L1) so the repo can be loaded ergonomically outside this checkout. A minimal `pi` manifest plus `pi-package` keywords would align `ozcar` with current Pi package conventions while preserving the project-local `.pi/` workflow.
- Add targeted regression coverage for the two missing seams above: one long-focus collision test around `deriveAuditId()` and one package/cross-project usage proof that documents the intended way to load `ozcar` when auditing a different repo.

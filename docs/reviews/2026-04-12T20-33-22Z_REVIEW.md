## GAPS
- Phase 2 acceptance explicitly requires `/reload` to keep the extension and its resources coherent (`PLAN.md:167-170`), but the slice never proves that behavior. The recorded validation stops at startup-only checks (`docs/execs/2026-04-12T20-08-04Z_EXEC.md:20-23`), and the automated tests only exercise the `resources_discover` hook for `reason: "startup"` plus direct command handler behavior (`tests/pi-extension.test.ts:64-119`). There is no reload-reason test and no Pi-side reload round-trip proof.
- The recorded `get_commands` proof overstates what it proves. Pi already auto-discovers project prompt templates from `.pi/prompts/*.md` and skills from `.pi/skills/` (`/tmp/pi-mono-ozcar-20260412/packages/coding-agent/docs/prompt-templates.md:9-17`, `/tmp/pi-mono-ozcar-20260412/packages/coding-agent/docs/skills.md:24-39`), so `docs/execs/2026-04-12T20-08-04Z_EXEC.md:22-23` does not isolate whether `resources_discover` is responsible for `ozcar-audit` and `skill:ozcar-audit`. I reran a stronger isolated proof with `pi --mode rpc --offline --no-session --no-extensions --no-skills --no-prompt-templates -e ./.pi/extensions/ozcar/index.ts`, and it did return `ozcar`, `ozcar-audit`, and `skill:ozcar-audit`, which shows the implementation is correct but the persisted proof is weaker than it should be.

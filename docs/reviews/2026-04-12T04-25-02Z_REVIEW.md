# REVIEW

## VERDICT
FAIL
Phase 6 lands the happy-path live seam, but the current slice still has one artifact-corrupting contract bug in live planning and one operational diagnosis bug in `doctor`.

## BUGS
- High: Live plan output can contain duplicate `scanId` values and the contract accepts them (`src/contracts/plan.ts:17-33`, `src/prompts/planner.md:19-23`). `runScanPhase()` persists each scan under `loops/<loop>/scans/<scanId>` (`src/phases/scan.ts:82-127`), so duplicate ids overwrite the same request, prompt, output, and provider artifact tree. I reproduced this against the built code: a live plan with two `scanId: "0001"` entries produced two `ScanPhaseResult`s with the same `scanRoot`, so the durable store no longer matches the accepted plan.
- Medium: `doctor` misdiagnoses preflight-failed live runs as missing `plan.json` instead of surfacing the persisted preflight failure. `inspectRunHealth()` jumps straight to `assertPlanGate()` whenever the run is in `plan` or later (`src/store/run-health.ts:66-75`). A reproduced `dist/cli.js run` failure with fake Claude auth missing persisted `provider/preflight.json` and `provider.preflight.failed`, but `dist/cli.js doctor` only reported `plan.gate` missing `loops/0001/plan.json`.

## GAPS
- The Phase 6 test suite does not cover the terminal parser-failure branch even though the runtime has explicit parse-error retry and `ProviderExecutionError` handling (`src/providers/runtime.ts:279-367`). `tests/phase6.test.ts` covers preflight failure, transient retry, happy-path live run/resume/replay/rebuild/doctor, and dry-run preservation (`tests/phase6.test.ts:43-407`), but no case asserts persisted parse-failure artifacts or terminal parse-failure behavior.
- The Phase 6 tests also do not exercise `doctor` against a preflight-failed live run. The suite checks a healthy live `doctor` path (`tests/phase6.test.ts:210-307`) and a separate preflight failure (`tests/phase6.test.ts:310-350`), which left the misleading `doctor` output above unvalidated.

## ANTI-PATTERNS
- The live plan seam currently trusts provider-emitted `scanId` values as filesystem keys without a uniqueness gate. That lets model output shape durable storage directly instead of being normalized or rejected before persistence.

## IMPROVEMENTS
- Reject duplicate `scans[*].scanId` values in the live plan contract or plan gate before any scan directories are created, and add a deterministic seam test for that rejection.
- Teach `inspectRunHealth()` to surface run-level preflight failures from `provider/preflight.json` or the recorded preflight events before requiring `loops/<loop>/plan.json`.
- Add one deterministic parser-failure test that proves persisted attempt artifacts and the terminal `ProviderExecutionError` contract.

## BUGS
- High: [confirmed-findings.ts](/Users/x/oz/ozcar/src/reports/confirmed-findings.ts#L25) writes `finding.summary`, `triage.rationale`, and `validation.rationale` directly into single-line bullets. The contracts in [finding.ts](/Users/x/oz/ozcar/src/contracts/finding.ts#L16), [triage.ts](/Users/x/oz/ozcar/src/contracts/triage.ts#L10), and [validation.ts](/Users/x/oz/ozcar/src/contracts/validation.ts#L10) allow valid multiline strings, so valid JSON can generate malformed `confirmed-findings.md`. I reproduced this end to end with built `runSummarizePhase()`, which emitted broken bullets for newline-bearing fields.
- Medium: [summarize.ts](/Users/x/oz/ozcar/src/phases/summarize.ts#L64) parses `finding.json` before calling [assertValidationGate()](/Users/x/oz/ozcar/src/gates/validate.ts#L40). If a validated artifact is malformed, rebuild fails with a raw parser exception instead of the deterministic, actionable `ValidationGateError` used by the rest of the Phase 3 validation seam. I reproduced this against the built code with an invalid `finding.json`.

## GAPS
- [phase3.test.ts](/Users/x/oz/ozcar/tests/phase3.test.ts#L309) covers the validated-only happy path, but it does not exercise summarize-time malformed validated artifacts or multiline report fields. That missing coverage is why both report-path regressions above can ship while `npm test` still passes.

## ANTI-PATTERNS
None

## IMPROVEMENTS
- Route summarize-time artifact loading entirely through the validation gate, or derive `scanId` and `findingId` from the validated artifact directory name before entering the gate, so rebuild failures stay typed and deterministic.
- Normalize or escape multiline report fields before emitting Markdown so the JSON-first contracts do not implicitly depend on single-line summaries or rationales.

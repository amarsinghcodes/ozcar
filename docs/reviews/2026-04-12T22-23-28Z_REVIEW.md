# REVIEW

## VERDICT
PARTIAL
Phase 4 lands the intended JSON-first storage and mechanical rebuild surface, and the recorded validation still passes, but the current implementation does not fully satisfy the determinism and storage-integrity claims.

## BUGS
- High: The store reader trusts embedded ids instead of the requested audit path, and the rebuild path is then derived from `snapshot.audit.auditId`. A corrupted `audit.json` inside one audit root can redirect rebuild output into a different audit root, and renamed finding directories still pass as fresh because directory names are never validated. Evidence: [src/artifacts/store.ts](/Users/x/oz/ozcar/src/artifacts/store.ts#L77), [src/artifacts/store.ts](/Users/x/oz/ozcar/src/artifacts/store.ts#L101), [src/artifacts/rebuild.ts](/Users/x/oz/ozcar/src/artifacts/rebuild.ts#L60). I reproduced both cases during review: renaming `findings/reentrant-withdraw/` to `findings/mismatched-directory-name/` still let `readAuditArtifactSnapshot()` return `reentrant-withdraw` and `inspectAuditArtifactFreshness()` report all artifacts fresh; changing the embedded `auditId` to `authorization-audit` caused `rebuildAuditArtifacts({ auditId: "payments-vault" })` to write reports under `.ai-auditor/audits/authorization-audit/`.
- Medium: Timestamp handling is parseable-string based rather than canonicalized, so semantically identical timestamps can produce different export bytes. `generatedAt` preserves whichever equivalent timestamp string appears first among the latest values, which breaks deterministic `exports/findings.json`. Evidence: [src/contracts/run.ts](/Users/x/oz/ozcar/src/contracts/run.ts#L8), [src/reports/export-findings.ts](/Users/x/oz/ozcar/src/reports/export-findings.ts#L38), [src/artifacts/store.ts](/Users/x/oz/ozcar/src/artifacts/store.ts#L89). I reproduced this with equivalent instants encoded as `2026-04-12T20:08:00.000Z` and `2026-04-12T16:08:00-04:00`; the rendered export JSON differed.

## GAPS
- The Phase 4 seam still materializes from an arbitrary snapshot file on disk and only checks audit-id alignment with the active branch. It does not derive findings from Pi-backed stored state or validate provenance beyond schema shape, so the implementation still falls short of “materialize durable repo-owned JSON artifacts from approved Pi-backed audit state” even though it correctly avoids transcript scraping. Evidence: [.pi/extensions/ozcar/commands/export.ts](/Users/x/oz/ozcar/.pi/extensions/ozcar/commands/export.ts#L117), [.pi/extensions/ozcar/commands/export.ts](/Users/x/oz/ozcar/.pi/extensions/ozcar/commands/export.ts#L123), [.pi/extensions/ozcar/commands/export.ts](/Users/x/oz/ozcar/.pi/extensions/ozcar/commands/export.ts#L147).
- The Phase 4 tests are not sufficient to prove the determinism and integrity claims made by the execution artifact. They permute finding order, but they do not cover canonical timestamp normalization, audit-root and finding-directory identity checks, or missing-artifact freshness cases. Evidence: [tests/audit-artifacts.test.ts](/Users/x/oz/ozcar/tests/audit-artifacts.test.ts#L104), [tests/audit-artifacts.test.ts](/Users/x/oz/ozcar/tests/audit-artifacts.test.ts#L136), [tests/pi-extension.export.test.ts](/Users/x/oz/ozcar/tests/pi-extension.export.test.ts#L30).

## ANTI-PATTERNS
None

## IMPROVEMENTS
- Canonicalize timestamps on parse or write, then derive `generatedAt` from the normalized values so equivalent instants collapse to one export representation.
- Validate that `audit.json`, `scope.json`, and each finding bundle match both the requested audit root and the containing finding directory before rebuild or freshness inspection.
- Add explicit regression tests for corrupted store paths, equivalent timestamp encodings, and missing `summary.md`, `confirmed-findings.md`, or `exports/findings.json`.

# REVIEW

## VERDICT
PARTIAL
`ozcar` can reach CritFinder-level usability for Claude Code and Codex without abandoning its current TypeScript, artifact-first design, but it needs one missing seam first: a real provider runner that executes authenticated agent calls instead of stopping at provider detection and dry-run artifact synthesis.

- The current provider boundary is metadata-only. [base.ts](/Users/x/oz/ozcar/src/providers/base.ts#L18) through [base.ts](/Users/x/oz/ozcar/src/providers/base.ts#L93) define capabilities, default models, and PATH detection, while [codex.ts](/Users/x/oz/ozcar/src/providers/codex.ts#L3) and [claude.ts](/Users/x/oz/ozcar/src/providers/claude.ts#L3) stop at static provider metadata.
- The operational path still rejects live execution. [run.ts](/Users/x/oz/ozcar/src/commands/run.ts#L43) requires `--dry-run`, and [scan.ts](/Users/x/oz/ozcar/src/phases/scan.ts#L238) through [scan.ts](/Users/x/oz/ozcar/src/phases/scan.ts#L260) only build synthetic dry-run output.
- CritFinder’s ergonomic win is a thin live-execution seam, not deeper architecture. [provider.sh](/Users/x/oz/apprentice-victor/CritFinder/lib/provider.sh#L253) runs explicit preflight, and [provider.sh](/Users/x/oz/apprentice-victor/CritFinder/lib/provider.sh#L261) retries provider-backed agent execution behind one `run_agent` entrypoint.
- Fresh `oss` analysis of `badlogic/pi-mono` shows that the reusable OAuth value is concentrated in `packages/ai`, not in the larger coding-agent stack. The generated report at [/tmp/oss-pi-mono/pi-mono_PATTERNS.md](/tmp/oss-pi-mono/pi-mono_PATTERNS.md#L58) correctly points at `packages/ai` and `packages/coding-agent/src/core` as the strongest reusable seams.

## BUGS
None

## GAPS
- `ozcar` has no provider execution contract analogous to CritFinder’s `run_agent`, so a logged-in `codex` or `claude` user still cannot perform a real audit run through the tool.
- `ozcar` has no auth surface of its own. [cli.ts](/Users/x/oz/ozcar/src/cli.ts#L13) exposes only `init`, `resume`, `status`, `run`, `replay`, `rebuild`, and `doctor`; there is no `login`, `logout`, `auth status`, or `models` flow.
- `ozcar` has no ergonomic first-run workflow that turns “I have a Claude/Codex account” into “audit this repo.” The user still has to know low-level flags such as `--scope`, `--scan-target`, and `--research-direction`.
- If `ozcar` wants native OAuth instead of “use the upstream CLI you already logged into,” it also lacks storage, refresh, and locking around OAuth credentials.

## ANTI-PATTERNS
- Depending on all of `pi-mono` would be the wrong shape for this repo. `PLAN.md` keeps `ozcar` TypeScript-only with one small provider interface, while `pi-mono`’s `packages/coding-agent` layer brings in sessions, TUI, extensions, and broader agent runtime concerns that `ozcar` does not need for parity.
- Adding OAuth before adding live execution would invert the dependency chain. CritFinder parity comes from executing agent calls ergonomically; auth only matters once `ozcar` can actually use the resulting credentials.

## IMPROVEMENTS
- Phase 1 should copy CritFinder’s thin execution seam into TypeScript: add a provider runner interface such as `plan(prompt, model)` and `scan(prompt, model)`; implement CLI-backed adapters that shell out to authenticated `codex` and `claude`; and make `run`/`resume` call that seam instead of the current dry-run-only path.
- Phase 1 should also copy CritFinder’s preflight ergonomics: automatic provider detection, explicit preflight failure with install/login guidance, default per-phase models, retry policy, and raw stdout/stderr artifact capture for replay and debugging.
- Phase 2 should add an `ozcar audit <target>` style wrapper and a minimal auth UX. The thin CLI should infer scope defaults from the repo, expose `ozcar doctor --auth`, and surface “detected codex/claude, logged in or not, default model” before the run starts.
- `@mariozechner/pi-ai/oauth` is a credible source for native OAuth support if `ozcar` wants its own login flow. The package is published as MIT under [`@mariozechner/pi-ai`](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/package.json#L2) and exports a dedicated `./oauth` entrypoint at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/package.json](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/package.json#L49). Its reusable core is the provider registry in [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/index.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/index.ts#L42), the callback-driven provider interface in [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/types.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/types.ts#L26), and the locked `auth.json` storage pattern in [/tmp/repo-patterns-badlogic-pi-mono/packages/coding-agent/src/core/auth-storage.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/coding-agent/src/core/auth-storage.ts#L45).
- Native OAuth should remain optional. For immediate parity, prefer “reuse already-authenticated CLIs” first. If you later want `ozcar login claude` and `ozcar login codex`, copy or vendor the small OAuth/auth-storage shapes from `pi-ai` rather than importing the whole coding-agent package.
- If you choose direct API mode, `pi-ai` already demonstrates the hard parts: Anthropic OAuth uses callback-server plus manual-paste login at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/anthropic.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/anthropic.ts#L227), and OpenAI Codex OAuth does the same with token refresh plus `accountId` extraction at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/openai-codex.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/utils/oauth/openai-codex.ts#L289). `pi-ai` also shows provider-specific request shaping for OAuth-backed direct calls, including Claude Code identity headers at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/providers/anthropic.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/providers/anthropic.ts#L565) and Codex account-aware headers at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/providers/openai-codex-responses.ts](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/src/providers/openai-codex-responses.ts#L876).
- If you adopt `@mariozechner/pi-ai` directly, budget for one integration wrinkle: `ozcar` currently builds CommonJS output as shown in [dist/cli.js](/Users/x/oz/ozcar/dist/cli.js#L1), while `@mariozechner/pi-ai` is ESM-only at [/tmp/repo-patterns-badlogic-pi-mono/packages/ai/package.json](/tmp/repo-patterns-badlogic-pi-mono/packages/ai/package.json#L5). That is manageable with dynamic import or a repo-wide ESM move, but it should be a conscious choice.

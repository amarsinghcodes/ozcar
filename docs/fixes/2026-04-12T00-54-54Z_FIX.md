# FIX

## PLAN

- Root cause: `runScanPhase()` rebuilds provider name, selection, and model from live scan inputs instead of treating the `plan.json` provider snapshot as the canonical cross-phase contract.
- Root cause: `assertScanGate()` validates only run, loop, and scan identity, so a separately produced `request.json` can drift from `plan.json` without deterministic rejection.
- Repair the scan phase to preserve the planned provider snapshot in prompts and request artifacts, and reject provider/model/selection mismatches before writing scan artifacts.
- Extend the scan gate to read the loop plan artifact and fail deterministically when `request.json` no longer matches the planned provider snapshot.
- Add focused seam tests for preserved provider continuity and deterministic gate rejection on cross-phase provider drift, then rerun the Phase 2 validation surface.

## CHANGES

- Updated [src/phases/scan.ts](/Users/x/oz/ozcar/src/phases/scan.ts) so the scan phase now treats `options.plan.provider` as the persisted provider snapshot for `prompt.md` and `request.json`, and added a deterministic `ScanPhaseProviderMismatchError` when a separately invoked scan tries to change provider name, selection mode, or model from the planned contract.
- Updated [src/gates/scan.ts](/Users/x/oz/ozcar/src/gates/scan.ts) so the scan gate now reads the sibling `plan.json`, validates its run and loop identity, and rejects `request.json` when `provider.name`, `provider.selection`, `provider.model`, or `provider.available` drift from the planned snapshot.
- Expanded [tests/phase2.test.ts](/Users/x/oz/ozcar/tests/phase2.test.ts) to assert direct equality between the planned and scanned provider snapshots in the normal dry-run seam, preserve a custom planned model across a separately invoked scan, reject a runtime provider mismatch before scan artifacts are written, and exercise deterministic gate failure for a request that drifts from `plan.json`.
